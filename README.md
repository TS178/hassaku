# 神輿リアルタイム位置表示システム

祭りの神輿14基の現在地を、見物客がスマホ／PCのブラウザだけで確認できるシステムです。
**専用アプリ不要・サーバー費用ほぼ0円・プログラミング初心者でも保守可能**を目標に構成しています。

---

## 1. 全体構成図

```
【神輿側】14基                【クラウド】             【閲覧者】100〜500人
┌──────────────┐          ┌────────────────┐        ┌──────────────┐
│ 中古Android   │  30秒毎   │ Google Apps      │  30秒毎 │ GitHub Pages  │
│ + SIM         │ ───POST──▶│ Script (無料)     │◀──GET── │ + Leaflet     │
│ 送信ページを   │  位置情報  │  ├ 位置を受信      │  最新位置│ 地図に14基表示 │
│ ブラウザで開く │          │  └ Spreadsheetに保存│        │ 一覧・検索・   │
│ (Wake Lock)   │          │    (＝データ倉庫)   │        │ 通信断表示     │
└──────────────┘          └────────────────┘        └──────────────┘
```

- **送信も閲覧も同じGitHub Pages**（HTTPSでGPS利用が可能）
- サーバーは Google Apps Script（GAS）1つだけ。データはGoogleスプレッドシートに貯まる
- スプレッドシートを開けば全神輿の生データが目で見えるので、初心者でも状況把握・手直しが可能

## 3. 想定コスト

**月額の固定サーバー費用は 0円。** かかるのは端末とSIMの実費のみです。

| 項目 | 単価の目安 | 14台合計 |
|---|---|---|
| 中古Androidスマホ | 3,000〜8,000円/台（買い切り） | 約4〜11万円 |
| SIM（データ専用） | 下記2パターン参照 | 下記参照 |
| サーバー（GAS） | 0円 | 0円 |
| GitHub Pages | 0円 | 0円 |

### SIMの2パターン

- **月額契約（データ専用・小容量）**：1回線 約300〜1,000円/月 × 14 = **約4,200〜14,000円/月**
- **プリペイド／日数制（祭り当日のみ）**：1回線 数百円〜1,500円程度 × 14 = **約数千〜2万円/イベント**

### 祭り当日のみ運用した場合

- サーバー費用：**0円**（GASは使った分だけ、無料枠内）
- 通信費：**プリペイドSIM代のみ**（例：14回線×約1,000円＝約14,000円/2日間）
- 端末は使い回せるため、2年目以降は**実質SIM代のみ**で運用可能

### データ通信量の目安（1台）
1回の送信＝約0.3KB。30秒毎×10時間＝約1,200回＝**1日あたり約0.5MB**。
14台でも1日10MB未満。小容量SIMで十分です。

## 4. 位置情報送信方法の比較

| 項目 | Google Apps Script | Firebase | Supabase | Cloudflare Workers |
|---|---|---|---|---|
| 無料枠 | ◎ 実質無制限（個人利用） | ○ 10GB/月・同時100接続 | ○ 500MB DB・帯域あり | ○ 10万req/日 |
| 実装難易度 | ◎ 最も簡単（コード貼るだけ） | △ SDK設定が必要 | △ SQL/認証設定が必要 | △ KV等の知識が必要 |
| リアルタイム性 | ○ ポーリング（30秒毎で十分） | ◎ 即時プッシュ可 | ◎ 即時プッシュ可 | ○ ポーリング |
| 同時14台の書き込み | ◎ Lockで安全に処理 | ◎ 得意 | ◎ 得意 | ◎ 得意 |
| 保守性（初心者） | ◎ データがシートで見える | △ 管理画面に慣れが必要 | △ DB画面に慣れが必要 | △ CLI/設定が中心 |
| クレカ登録 | 不要 | 不要（無料枠） | 不要 | 不要 |

**結論：** 更新間隔30秒・14台という規模では「リアルタイム性」の差は体感できません。
**初心者の保守性と完全無料**を最優先し、本システムは **Google Apps Script** を採用します。
（閲覧者が常時500人を超え、より余裕が欲しい場合はFirebaseへの差し替えを検討）

## 5. 推奨構成

- **サーバー：Google Apps Script（GAS）+ スプレッドシート**
- **公開：GitHub Pages**
- **地図：Leaflet + OpenStreetMap**
- **送信：ブラウザのGeolocation API（専用アプリ不要）**
- **負荷対策：GAS側で15秒キャッシュ**（閲覧者が増えてもシート読み取りは最大4回/分に抑制）

この構成なら、ソースはHTML/CSS/JSと1枚のGASコードだけ。編集箇所は `js/config.js` に集約されています。

---

## 6. サンプルコード（同梱ファイル）

すべてコピペで動きます。編集するのは基本 `js/config.js` の①〜⑦だけです。

| ファイル | 役割 | 実装した要件 |
|---|---|---|
| `js/config.js` | **管理用設定ファイル** | 神輿ID・名前・色、GAS URL、更新間隔、通信断秒数 |
| `sender/index.html` | **GPS送信ページ** | 神輿選択・GPS取得・30秒送信・画面常時点灯・送信ログ |
| `gas/Code.gs` | サーバー | 受信保存・最新位置のJSON配信・15秒キャッシュ・排他制御 |
| `index.html` | **Leaflet表示ページ** | 地図・14基表示・検索窓・一覧パネル |
| `js/map.js` | 閲覧ロジック | **神輿一覧表示・更新時刻表示・通信断判定**・速度表示・検索 |
| `css/style.css` | 見た目 | 藍×朱の和風・屋外/スマホで見やすい高コントラスト |

主要機能の実装ポイント：
- **神輿一覧**：右（PCは横／スマホは下シート）に14基を色つきで一覧。タップで地図が該当神輿へ移動
- **更新時刻表示**：「◯秒前 / ◯分前」＋最終更新時刻。ポップアップにも表示
- **通信断判定**：`OFFLINE_SEC`（初期120秒）を超えて更新がなければアイコンを灰色にし「通信断」表示。一度も受信がなければ「未受信」
- **速度表示**：端末が返す `speed`(m/s) を km/h に換算（取得できない端末は「取得不可」）

## 8. セキュリティ対策

本システムは「位置を見せる」ことが目的のため、閲覧は誰でも可。守るべきは**なりすまし送信**と**個人情報**です。

- **合言葉（API_KEY）**：送信時に合言葉を照合。知らない人はデータを書き込めない
  （`config.js` と `Code.gs` の両方を同じ文字列に。祭りごとに変更する）
- **送信ページの周知を絞る**：`sender/` のURLは運営者だけに共有する
- **個人情報を送らない**：送るのは神輿の位置のみ。担ぎ手個人を特定する情報は扱わない
- **保存範囲を限定**：シートには最新の1行のみ上書き保存（過去の移動履歴を溜め込まない）
- **合言葉は完全な認証ではない**点に注意：ページを見れば合言葉は読めます。**荒らし対策の第一段階**と位置づけ、当日はURLを限定共有し、異常な値が入ったらシートで手修正・該当行削除で対応します
- **公開後の後始末**：祭り終了後はGASのデプロイを「アーカイブ」して受付を止める

> より強固にするなら、GAS側で「会場の緯度経度から大きく外れた座標は捨てる」バリデーションを追加できます。

## 9. 当日運用マニュアル

### 前日まで
1. 14台すべて充電満タン
2. 各端末でモバイル通信ONを確認、ブラウザで送信URL `…/sender/` を開いてブックマーク
3. GASのURLとAPI_KEYが本番用になっているか、実際に1台で送信テスト（閲覧ページに出るか確認）

### 当日 出発前（各神輿）
1. 端末を神輿の**空が見える位置**に固定（GPS精度のため。金属で覆わない）
2. モバイルバッテリーを接続
3. 送信ページを開き、**担当の神輿を選択 →「送信開始」**
4. 位置情報の許可を「許可」。画面が「🟢 送信中」になればOK
5. 画面は消さない（自動で点灯維持しますが、念のためスリープ時間も長めに設定）

### 巡行中（本部）
1. PCまたはタブレットで閲覧ページを開き、一覧の「通信断」件数を監視
2. 通信断が出た神輿は担当へ連絡（端末の状態・電池・電波を確認）

### 終了後
1. 各端末で「停止」を押す
2. GASのデプロイをアーカイブ（受付停止）
3. スプレッドシートの `positions` シートは翌年まで残してOK（上書き運用のため肥大化しない）

## 10. 障害対策

| 症状 | 主な原因 | 対処 |
|---|---|---|
| 地図に出てこない | 送信ページで「開始」してない／位置情報を拒否 | 送信ページを開き直し「許可」→「送信開始」 |
| 「通信断」になる | 圏外・電池切れ・画面スリープ・端末発熱 | 電波の良い場所へ／充電/交換／直射日光を避ける |
| 位置がずれる | ビル街・地下でGPS精度低下 | 空の見える場所に端末を移す。数分で復帰することが多い |
| 全神輿が同時に消えた | GAS/ネットワーク障害 | 閲覧側は「接続できません」を表示し自動再試行。数分待つ |
| 速度が「取得不可」 | 端末が速度を返さない仕様 | 異常ではない（位置は正常に取得できていればOK） |
| 送信できない | 合言葉不一致 | `config.js` と `Code.gs` のAPI_KEYを一致させ再デプロイ |

**予備策**：
- 予備のスマホ＋SIMを1〜2台用意し、故障時に神輿IDを選び直して即交代
- モバイルバッテリーは各神輿に2個体制（1個は予備）
- 本部PCが不調でも、スマホから閲覧ページを開けば状況確認可能

## 11. GitHub Pages への公開手順

### A. Googleスプレッドシート＆GASの準備
1. Googleドライブで**スプレッドシートを新規作成**（名前は任意、例：神輿トラッカー）
2. 上部メニュー **拡張機能 → Apps Script**
3. 出てきたエディタの中身を全部消し、`gas/Code.gs` の内容を貼り付け
4. `API_KEY` を好きな合言葉に変更して保存（💾）
5. 右上 **デプロイ → 新しいデプロイ**
   - 種類（⚙）：**ウェブアプリ**
   - 実行するユーザー：**自分**
   - アクセスできるユーザー：**全員**
   - **デプロイ** →（初回は権限承認。「詳細」→「安全でないページに移動」で許可）
6. 表示された **ウェブアプリURL（…/exec）** をコピー

### B. 設定ファイルの編集
7. `js/config.js` を開き、
   - `GAS_URL` に手順6のURLを貼る
   - `API_KEY` を手順4と同じ合言葉にする
   - `MAP_CENTER` を会場の緯度経度に、`MIKOSHI` の名前を実際の神輿名に変更

### C. GitHubで公開
8. GitHubで**新しいリポジトリを作成**（Public）
9. `mikoshi-tracker` フォルダの中身一式をアップロード
   （画面上で「Add file → Upload files」にドラッグ＆ドロップでOK）
10. リポジトリの **Settings → Pages**
    - Source：**Deploy from a branch**
    - Branch：**main** ／ フォルダ：**/(root)** → **Save**
11. 1〜2分待つと公開URLが表示される
    - 閲覧：`https://ユーザー名.github.io/リポジトリ名/`
    - 送信：`https://ユーザー名.github.io/リポジトリ名/sender/`

### D. 動作確認
12. 送信URLをスマホで開き、神輿を選び「送信開始」→ 閲覧URLに現れれば成功

## 12. 最終的な推奨構成（結論）

**「GitHub Pages（送信＋閲覧）＋ Google Apps Script ＋ スプレッドシート ＋ Leaflet」**

| 前提 | 本構成での対応 |
|---|---|
| 神輿14基 | `config.js` に14基定義済み。色・番号で区別 |
| 閲覧者100〜500人 | GAS側15秒キャッシュで読み取り負荷を抑制。500人超が想定される場合のみFirebaseへ差し替え検討 |
| 更新間隔30秒 | 送信・閲覧とも30秒（設定で変更可） |
| 祭り2日間 | 無料枠で完結。端末は使い回し |
| できるだけ無料 | サーバー0円。実費はSIMと端末のみ |
| 初心者でも保守可能 | 編集は`config.js`に集約。データはシートで目視・手修正可 |
| コピペで動く | 同梱コードをそのまま利用。修正は設定値のみ |
| HTML/CSS/JS優先 | フロントは純粋なHTML/CSS/JS。サーバーはGAS1枚のみ |

これが、**低コスト・低学習コスト・低運用リスク**のバランスが最も良い最適解です。
